Overview
This PRD defines a browser-based, mobile-first multiplayer photo puzzle game. Players
take turns setting photo-based puzzles for each other and guessing each other’s
puzzles, with one turn per day. The game is designed for small invite-based groups with
open-ended play and cumulative scoring.

Core Concept
Each day constitutes one turn. Every player in a group simultaneously acts as a setter
(creating one puzzle for the others) and a guesser (attempting to solve each other
player’s puzzle). For a group of N players, there are N concurrent rounds running at any
time.
A setter uploads a personal photograph and provides metadata (What, Who, Where,
When). They can optionally add visual and textual clues. Guessers see a limited view of
the photo and attempt to identify the metadata. Correct answers lock in points; incorrect
answers require the guesser to either reveal a clue (reducing available points) or
concede.

Gameplay Mechanics
Setting a Puzzle
The setter flow is a 6-step wizard:
1.​ Upload Subject Photo: The setter selects a photo from their device. On mobile,
this uses the native photo picker. On desktop, a standard file upload dialog. This
is the core image of the puzzle.
2.​ Define Sub-portions (optional): The setter can draw up to 3 rectangles on the
subject photo. These create cropped views that are revealed progressively to the
guesser. The first sub-portion is shown for free as the initial view.
3.​ Upload Clue Photos (optional): Up to 3 additional photographs that serve as
visual hints. These are separate images, not crops of the subject photo.
4.​ Enter Metadata: The setter provides answers for four fields. If a field is not
applicable (e.g. no people in the photo), the setter marks it as N/A. Fields marked
N/A are excluded from the guessing round and scoring.
5.​ Add Text Clues (optional): The setter can request AI-generated clues (the
system produces 5, the setter picks 0–3) and/or manually write up to 3 clues.
Total across AI-selected and manual cannot exceed 3.
6.​ Review & Submit: The setter sees a summary including the total points
available, all reveals, and the answers. They confirm and submit.

AI Clue Generation
When the setter taps the “Generate AI Clues” button, the system sends the subject
photo to a vision-capable LLM with a prompt requesting 5 varied, progressively helpful
clues. The clues should:
•​ Avoid directly naming the subject, location, or people
•​ Range from oblique hints to more direct ones
•​ Reference visual, historical, cultural, or geographic attributes
•​ Be phrased as single sentences
The setter selects 0 to 3 of the 5 generated clues. They can also write their own, up to a
combined total of 3 text clues.

Guessing a Puzzle
When a guesser opens a puzzle, they see the initial image:
•​ If sub-portions exist: The first sub-portion is displayed, with the rest of the
subject photo blurred out. This initial sub-portion is free (does not cost a reveal).
•​ If no sub-portions: The full subject photo is displayed.
The guesser enters answers for all non-N/A fields and submits. Each field is evaluated
independently:
•​ Correct: The field locks in immediately with points awarded at the current reveal
level. It is shown as solved and cannot be changed.
•​ Incorrect: The field remains unsolved. The guesser must choose to either reveal
one additional piece of information (any type) or concede.
One guess per field per reveal level. The guesser cannot re-submit answers for the
same field at the same reveal level. After submitting, if any fields are wrong, they must
reveal a clue or concede before guessing again.
Reveal choice: When revealing, the guesser chooses the type (next sub-portion, a clue
photo, or a text clue). The guesser picks which category but not which specific item;
reveals are sequential within each type.
Full photo: The full subject photo is only shown if the guesser concedes or runs out of
reveals. It is never shown while the game is active (if sub-portions were provided).
Concession
The guesser can concede at any point. On concession:
•​ Points for any previously correct fields are kept
•​ Unsolved fields score 0
•​ The full subject photo is revealed
•​ All correct answers are shown

Missed Turns
If a guesser does not play by 06:00 the following day (configurable), they score 0 for
that puzzle. The puzzle is then resolved and the setter receives maximum points for that
guesser.

Scoring System
Points Pool Calculation
The total points available for a puzzle are:

Total Points = 25 + (Number of Reveals × 10)
Where “Number of Reveals” is the sum of sub-portions + clue photos + text clues.

Guesser Scoring
Points are divided equally across all non-N/A fields. With the example above and 3
non-N/A fields, each field is worth 75 ÷ 3 = 25 points at the first guess level.
Each reveal reduces the total by 10 points:
•​ First guess (0 reveals used): 75 points total, 25 per field
•​ After 1 reveal: 65 points total, ~21.7 per field
•​ After 2 reveals: 55 points total, ~18.3 per field
•​ After all 5 reveals: 25 points total, ~8.3 per field
A field’s points are locked in at the level when the correct answer is submitted. Different
fields can be scored at different reveal levels.

Answer Evaluation
All answer evaluation is performed by an LLM to allow for fuzzy, natural-language
matching. The system sends the setter’s answer and the guesser’s answer as a pair,
along with the field type, and asks for a binary match/no-match decision.

Matching Rules by Field
What
The broadest matching. Accept synonyms, reasonable descriptions, and alternate
names. “Tower Bridge” should match “the bridge with two towers in London”. The LLM
should evaluate semantic equivalence, not string similarity.
Who
Name matching with awareness of common nicknames, diminutives, and
formal/informal variants. Examples:
•​ Jess ↔ Jessica
•​ Mike ↔ Michael
•​ Dave ↔ David
•​ Bob ↔ Robert
•​ Full name match if only first name given or vice versa
Where
Geographic containment and equivalence. A more specific answer should match a more
general one:
•​ "North London" matches "London"
•​ "Manhattan" matches "New York"
•​ "Tuscany" matches "Italy"
•​ But "Paris" does not match "France" (too vague — the LLM should use
judgement)
When
Year only. Exact numeric match. The LLM should extract the year from both answers
and compare. “2019” matches “2019”. “Last summer” does not match (must be a year).
LLM Evaluation Prompt
The evaluation prompt should be structured to return a deterministic result. Suggested
approach:
•​ System prompt establishing the role as a quiz answer evaluator
•​ Include field type (What/Who/Where/When) and corresponding matching rules

Image Handling
•​ Upload: Subject photos and clue photos saved to the local filesystem under
a structured uploads/ directory (e.g. uploads/{groupId}/{puzzleId}/). Strip
EXIF on upload (except GPS and date for pre-fill) for privacy.
•​ Sub-portions: Stored as percentage-based coordinates. The frontend renders
the crop at display time. No server-side image processing required.
•​ Transformation: Use Sharp (Node.js) for server-side thumbnail generation
and WebP conversion on upload.
•​ Size limits: Max 10MB per image. Resize on upload to max 2000px on longest
edge.
•​ Serving: Images served as static files via Next.js public directory or a
dedicated /api/images/:path route with appropriate cache headers.

Real-time Updates
Server-Sent Events (SSE) via Next.js API routes for:
•​ Notifying setter when a guesser submits/reveals/concedes
•​ Updating leaderboard in real-time
•​ Alerting users when a new puzzle is available
Alternatively, short-polling (every 10–30 seconds) can be used as a simpler
fallback if SSE adds too much complexity for V1.

Background Jobs
•​ Turn expiry: Scheduled via node-cron (or a Next.js cron API route on
Vercel) to run daily at the configured deadline hour. Mark all active
GuessSessions as "expired" with score 0. Award full points to setters.
•​ Score rollup: After each puzzle completion, update
GroupMembership.total_score via application-level logic in the API route.

Security
•​ Authentication: All API endpoints require authenticated user
•​ Authorisation: Application-level middleware enforces access control. Users
can only access puzzles in their groups. Guessers cannot see answers or other
guessers' activity. Setters can see all guesser activity for their puzzles.

group membership. EXIF GPS data stripped after extraction.
•​ Answer security: Answers are never sent to the guesser’s client. All evaluation
happens server-side via LLM.
•​ Input validation: All user inputs sanitised. Image type and size validated
server-side.

UX Flows
First-Time User
1.​ Land on home screen (not logged in)
2.​ Sign up / log in (magic link or social via NextAuth.js)
3.​ See empty dashboard with “Create Group” or “Join Group”
4.​ Create group → get invite link to share
5.​ Or: tap invite link → join group → see dashboard

Daily Loop (Setter)
1.​ Open app → dashboard shows “Set today’s puzzle” prompt
2.​ Tap → enter 6-step setter wizard
3.​ Upload photo → optionally add sub-portions → optionally add clue photos
4.​ Enter What/Who/Where/When (or N/A)
5.​ Optionally generate/write clues
6.​ Review and submit
7.​ Return to dashboard → see pending guessers

Daily Loop (Guesser)
1.​ Open app → dashboard shows puzzles to solve
2.​ Tap a puzzle → see initial image
3.​ Enter guesses for all fields → submit
4.​ See results: correct fields lock in (green), wrong fields shown (red)
5.​ Choose: reveal a clue (pick type) or concede
6.​ If reveal: see new info, guess again
7.​ Repeat until all correct, conceded, or out of clues
8.​ See final score and setter’s score

Setter Activity View
After submitting a puzzle, the setter can monitor guesser activity:
•​ Which guessers have started, are in progress, or have finished
•​ For each guesser: their guess history per field, which reveals they chose, their
final score
•​ The setter’s points earned from each guesser
•​ This view updates in real-time via SSE or polling

Future Considerations
The following are explicitly out of scope for V1 but should be considered in the
architecture:
•​ Push notifications: Remind players to set/guess before deadline. Architecture
should support adding web push or native notifications later.
•​ Streaks: Track consecutive days of play. Display on profile/leaderboard.
•​ Photo moderation: Automated NSFW detection on uploaded images. Flag for
review.
•​ Puzzle archive: Browse past puzzles and their solutions. “Replay” mode for new
group members.
•​ Group settings: Configurable deadline hour, timezone, weekend skips.
•​ Multi-language: LLM evaluation already handles this implicitly; UI localisation
would need separate work.
•​ Native apps: If the PWA approach has limitations (camera access, push),
consider React Native or Capacitor wrappers.
